# 模块化数学知识树设计文档

> 状态：待用户审阅
> 日期：2026-06-21
> 背景：现有知识树按"年级→学期→单元→课时"组织，不利于大模型按"数学概念"判断题目归属；改用"模块化主题体系"

## 1. 背景与问题

### 1.1 现状

当前 `preset_knowledge.py` 按**北师大版教材章节**组织：

```
数学
└─ 一年级
   └─ 一年级上册
      └─ 第一单元 · 生活中的数
         ├─ 10 以内数的认识
         ├─ 0 的认识
         ├─ 10 以内数的比较大小
         ├─ 10 以内数的顺序与位置
         ├─ 10 以内数的组成
         └─ 10 以内数的加减法（一）
```

共 492 个节点、4 层结构。

### 1.2 痛点

1. **场景化命名**让大模型难归类
   - "数松果（5 的乘法口诀）"、"买铅笔（十几减 9）"——名字含生活场景，题目里看不到"松果"就匹配不上
2. **跨年级主题分散**
   - "乘法口诀"在二年级上、"分数乘法"在五年级下、"百分数应用"在六年级上——大模型难以聚合同一主题
3. **粒度参差**
   - 有的"整理房间"没有具体数学概念，有的"加法交换律和结合律"过于细分
4. **4 层冗余**
   - "年级→学期→单元"三级对大模型是无用噪声

### 1.3 目标

用**模块化主题体系**重塑知识树，让大模型能从**题目内容**（而非章节）匹配到**数学概念**。

## 2. 设计方案

### 2.1 整体结构

**3 层（顶层模块 → 子主题 → 知识点），**沿用《义务教育数学课程标准》四大领域。

```
数学
├─ 数与代数
│  ├─ 数的认识
│  │  ├─ 10 以内数的认识
│  │  ├─ 11~20 各数的认识
│  │  └─ ...
│  └─ 数的运算
│     ├─ 10 以内加减法
│     ├─ 20 以内进位加法
│     └─ ...
├─ 图形与几何
│  └─ ...
├─ 统计与概率
│  └─ ...
└─ 综合与实践
   └─ 解决问题（应用题）
```

预估节点数：~90 个（vs 现有 492 个）。

### 2.2 顶层 4 大模块

| 模块 | 含义 | 典型子主题 |
| --- | --- | --- |
| **数与代数** | 数、运算、式、关系 | 数的认识 / 数的运算 / 式与方程 / 正比例与反比例 |
| **图形与几何** | 空间、形状、位置 | 图形的认识 / 图形的测量 / 图形的运动 / 图形与位置 |
| **统计与概率** | 数据、可能性 | 数据收集与整理 / 统计图表 / 平均数 / 可能性 |
| **综合与实践** | 应用、解决问题 | 解决问题（应用题） |

### 2.3 命名规范（**关键**）

**知识点名 = 数学概念名，不含生活场景**。

| 现有（场景化） | 新（概念化） | 说明 |
| --- | --- | --- |
| 数松果（5 的乘法口诀） | 5 的乘法口诀 | 去场景 |
| 买铅笔（十几减 9） | 20 以内退位减法（十几减 9） | 保留特例备注 |
| 整理房间 | （删除） | 非数学概念 |
| 美丽的田园 | （删除） | 纯应用题场景，归到"解决问题" |
| 反弹高度 | （删除） | 跨学科，无具体概念 |
| 加法交换律和结合律 | 加法交换律、加法结合律 | 拆为两个原子知识点 |
| 一图两式（一图四式） | 看图列式 | 去括号注释，命名精简 |

### 2.4 与大模型的协作

- **`match_knowledge.md` 提示词**保持现有逻辑（"从已有知识点列表中选择 1-3 个最匹配的"），不需要大改
- 节点名变成概念后，**大模型匹配准确率显著提升**（如题目"5×3=15"→直接命中"5 的乘法口诀"或"表内乘法"）
- 节点数从 492 降到 ~90，**大模型选择面更聚焦**，减少误判

## 4. 实施步骤

### 4.1 后端改动

1. **新增** `preset_module_based_math.py`（与 `preset_knowledge.py` 同目录）
   - 写入模块化结构
2. **修改** `preset_knowledge.py`：
   - `PRESET_BNUP_PRIMARY_MATH` 改为模块化版本
   - 保留 `PRESET_KNOWLEDGE_TREE = PRESET_BNUP_PRIMARY_MATH` 兼容旧调用
3. **修改** `app/services/knowledge_service.py`：
   - `initialize` 流程在 force=True 时，先**清空**该学科旧数据再导入新数据
4. **修改** `app/api/knowledge.py`：
   - `initialize_bnup_knowledge` 接口在 force=True 时**自动**清空旧数据
   - 增加"软删除"前的**题目引用检查**（如果叶子节点被题目绑定，提示用户先解绑）
5. **保持** `match_knowledge.md` 提示词不变（结构变化后大模型匹配反而更准）

### 4.2 前端改动

无需改动 UI（`KnowledgeGraphTree` 组件通用，渲染任意嵌套树）。

### 4.3 数据迁移

**只提供"force=true 清空重建"一种方式**（简单可靠）：
- 后端接口：调用 `POST /api/knowledge/initialize-bnup?force=true`
- 流程：清空 subject='数学' 的所有节点 + 关联 → 重新导入模块化结构
- 注意：清空前**备份**问题题目的 `question_knowledge` 关联（如有），清空后**重建**映射（如有题目已绑定叶子节点，迁移到最近的父节点）

## 5. 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 清空旧数据导致题目知识点引用丢失 | 迁移脚本：清空前**先解除所有绑定**，清空后**让用户重新触发大模型匹配** |
| 知识点名变化，大模型需要重新理解 | 节点名更"概念化"实际更易匹配，反而是降风险 |
| 旧数据被永久删除 | 提供"导出 JSON 备份"前置步骤（用户手动操作） |
| 节点数从 492 → 90，用户担心覆盖面不全 | 通过**用户手动新增**补充，组件已支持增删改移 |

## 6. 不在本次范围

- **不改前端 UI**（组件已支持任意嵌套）
- **不改大模型提示词**（结构变了反而匹配更准）
- **不引入版本管理/回滚**（一次性重构，靠 git 保护）
- **不跨学科**（本次只重构"数学"）

## 7. 验收标准

1. 重启后端，调用 `POST /api/knowledge/initialize-bnup?force=true` 后
   - 数据库 `knowledge_points` 表中 subject='数学' 的节点数 ~90
   - 无"年级/学期/单元"命名的节点
   - 顶层 4 个：数与代数 / 图形与几何 / 统计与概率 / 综合与实践
2. 前端系统设置 → 知识树管理 显示 4 大模块，可逐层展开
3. 用一道理科题测试大模型匹配，准确率显著高于旧结构（主观评估）
