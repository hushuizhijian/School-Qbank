# 二期优化计划

## 问题分析

当前 PDF 上传后解析失败，根本原因是 **MinerU 云端解析 + 分题切分耦合在同一个流程中**，导致：
- 解析失败时无法定位是 MinerU 解析阶段还是分题阶段出错
- 解析中无法切换分题方案，必须等整卷解析完才能重新分题
- 用户无法查看 MinerU 原始产物（word/html/json），只能看到 md/latex

---

## 优化一：MinerU 云端解析与分题切分拆分为两个独立页面

### 目标
将 `PaperUploadPage` 从"上传+解析+分题"一体式改为两步流程，降低耦合度，便于定位问题。

### 方案

#### Step 1 — MinerU 云端解析页 (`/papers/upload`)
- 保留：基础信息填写、文件上传
- 简化：去掉分题方案选择，只保留解析引擎配置
- 新增：`extra_formats` 多选（用户可勾选需要哪些格式产物）
- 流程：上传 PDF → 触发 MinerU 云端解析 → 轮询进度 → 解析完成后跳转分题页

#### Step 2 — 分题切分页 (`/papers/:id/split`)
- 新页面，从 MinerU 解析页完成后自动跳转过来
- 展示 MinerU 解析产物预览（md/html/latex 可选切换查看）
- 分题方案选择（request分题 / content分题 / LaTeX锚点分题）
- 分题结果编辑（沿用现有 SplitQuestionEditor）
- "重试解析"按钮回到 Step 1

### 后端改动
- 上传 API 不再自动触发分题，仅触发 MinerU 解析
- 新增 `/api/papers/{id}/split` 接口，在 MinerU 完成后手动触发分题
- `parse_service.py` 分拆为 `mineru_parse_service.py`（仅解析）和 `split_service.py`（仅分题）

---

## 优化二：MinerU 解析结果支持全部五种格式

### 现状
- `mineru_service.py` 只返回 `markdown` + `latex` + `content_list` + `images`
- `parse_service.py` 只保存 `.md` 和 `.tex` 到磁盘

### 目标
支持 MinerU SDK 能返回的全部五种格式：`markdown`、`latex`、`html`、`docx`、`content_list.json`

### 方案
1. **`MinerUParseResult`** 新增 `html` 和 `docx` 字段
2. **`mineru_service.py`** 解析时添加 `extra_formats: ["latex", "html", "docx"]`
3. **`parse_service.py`** 保存时额外保存：
   - `output.html`（HTML格式）
   - `output.docx`（Word格式）
   - 已有：`output.md`、`output.tex`、`content_list.json`
4. 前端新增文件预览切换（在分题页可切换查看不同格式的解析产物）

## 优化三：操作日志 SSE 面板增加可移动功能

### 现状
- `GlobalSSEPanel.tsx` 使用 `fixed bottom-4 right-4` 固定在右下角
- 支持折叠/展开，但不支持拖拽移动

### 方案
1. 在标题栏上添加 `mousedown` 事件监听
2. 拖拽时动态更新面板的 `left`/`top` 位置
3. 拖拽结束时保存位置到 `localStorage`
4. 组件初始化时从 `localStorage` 恢复位置
5. 双击标题栏恢复默认位置（右下角）

## 优化四：取消硬编码大模型，系统设置中增加大模型配置

### 现状
- 前端 `aiProviders.ts` 硬编码了 "智谱" 和 "DeepSeek" 两个供应商
- 后端 `ai_config_service.py` 已有完整 CRUD + 连接测试接口
- 后端 `parse_service.py` 硬编码查询 "智谱" provider 做分题

### 目标
在系统设置页增加"AI模型配置"标签页，用户可以自行添加/管理大模型，完全取消硬编码。

### 方案（参照 ragflow 的 factory 模式）

#### 前端改造
1. `SettingsPage.tsx` 新增 "AI模型" 标签页
2. 新建 `AIModelSettings.tsx` 组件：
   - 大模型供应商列表（表格展示：名称、API地址、模型列表、启用状态）
   - 添加供应商按钮 → 弹窗表单（名称、API地址、API Key、模型列表）
   - 每个供应商行：编辑、删除、连接测试、启用/禁用开关
   - 连接测试：发送简单 chat 请求验证 API Key 是否有效
3. 删除 `aiProviders.ts` 中的硬编码数据，改为从后端 API 获取

#### 后端改造
- 已有 `ai_config_service.py`（CRUD + 连接测试），无需大改
- 修改 `parse_service.py` 中的 `_get_zhipu_provider` → 改为 `_get_first_available_provider`，从数据库查第一个启用的供应商
- 修改 `parse_service.py` 中硬编码的模型名 `"GLM-4.7-Flash"` → 使用数据库中的 `model_list[0]`

## 风险评估

- **拆分页面**：需要修改路由，确保现有功能不受影响
- **五种格式**：html/docx 文件较大，注意磁盘空间
- **大模型配置**：需要测试连接功能，确保 API Key 加密存储
- **SSE 面板拖拽**：注意移动端兼容性（本项目为桌面端，风险低）
