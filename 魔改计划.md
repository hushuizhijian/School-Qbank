#### 2.2.2 适配器基类 `llm/providers/base.py`

```python
"""
LLM 适配器抽象基类

功能：定义所有 AI 供应商适配器的统一接口
输入参数：api_key / api_base / model
返回值：BaseLLMProvider 实例
使用场景：所有 LLM 适配器继承此基类
"""
from abc import ABC, abstractmethod


class BaseLLMProvider(ABC):
    """AI 服务商适配器基类 — 统一接口"""

    # 供应商名称（子类必须覆盖）
    provider_name: str = ""

    def __init__(self, api_key: str, api_base: str, model: str = ""):
        self.api_key = api_key  # API 密钥
        self.api_base = api_base.rstrip("/")  # API 基础地址
        self.model = model  # 模型名称

    @abstractmethod
    async def chat(
        self,
        system_prompt: str,
        user_prompt: str,
        temperature: float = 0.1,
        max_tokens: int = 4096,
    ) -> str:
        """通用对话接口，返回文本内容"""
        ...

    @abstractmethod
    async def chat_stream(
        self,
        system_prompt: str,
        user_prompt: str,
        temperature: float = 0.1,
        max_tokens: int = 4096,
    ):
        """流式对话接口，异步生成器"""
        ...

    async def health_check(self) -> bool:
        """健康检查：验证 API Key 是否有效"""
        ...

    async def list_models(self) -> list[dict]:
        """获取该供应商的可用模型列表"""
        ...
```

#### 2.2.3 模型工厂 `llm/factory.py`

```python
"""
LLM 模型工厂 — 统一调度所有 AI 供应商

功能：根据供应商名称动态创建适配器实例，管理适配器缓存
输入参数：provider_name / api_key / api_base / model
返回值：BaseLLMProvider 实例
使用场景：所有需要调用 LLM 的地方，通过工厂获取适配器
"""
import json
from pathlib import Path
from llm.providers.base import BaseLLMProvider
from llm.providers.openai import OpenAIProvider
from llm.providers.deepseek import DeepSeekProvider
from llm.providers.zhipu import ZhipuProvider


# 适配器注册表：供应商名称 → 适配器类
PROVIDER_REGISTRY: dict[str, type[BaseLLMProvider]] = {
    "OpenAI": OpenAIProvider,
    "DeepSeek": DeepSeekProvider,
    "智谱AI": ZhipuProvider,
}

# 适配器实例缓存
_provider_cache: dict[str, BaseLLMProvider] = {}

# 模型工厂配置路径
CONF_DIR = Path(__file__).parent / "conf"


def load_factory_config() -> dict:
    """加载 llm_factories.json 配置"""
    config_path = CONF_DIR / "llm_factories.json"
    with open(config_path, "r", encoding="utf-8") as f:
        return json.load(f)


def list_factories() -> list[dict]:
    """列出所有可用的 AI 供应商"""
    config = load_factory_config()
    return [
        {
            "name": f["name"],
            "logo": f["logo"],
            "tags": f["tags"],
            "url": f["url"],
            "status": f["status"],
        }
        for f in config["factory_llm_infos"]
    ]


def list_models(provider_name: str) -> list[dict]:
    """列出指定供应商的可用模型"""
    config = load_factory_config()
    for f in config["factory_llm_infos"]:
        if f["name"] == provider_name:
            return f["llm"]
    return []


def get_provider(
    provider_name: str,
    api_key: str = "",
    api_base: str = "",
    model: str = "",
) -> BaseLLMProvider | None:
    """获取 AI 供应商适配器实例（带缓存）

    Args:
        provider_name: 供应商名称（如 "DeepSeek"、"OpenAI"）
        api_key: API 密钥
        api_base: API 基础地址（可选，默认使用配置中的地址）
        model: 模型名称（可选，默认使用供应商的第一个模型）

    Returns:
        BaseLLMProvider 实例，如果供应商不存在或 API Key 无效则返回 None
    """
    # 检查供应商是否已注册
    if provider_name not in PROVIDER_REGISTRY:
        return None

    # 从配置中获取默认 API Base
    if not api_base:
        config = load_factory_config()
        for f in config["factory_llm_infos"]:
            if f["name"] == provider_name:
                api_base = f["url"]
                break

    # 缓存键
    cache_key = f"{provider_name}:{api_key}:{api_base}:{model}"

    if cache_key in _provider_cache:
        return _provider_cache[cache_key]

    # 创建适配器实例
    provider_cls = PROVIDER_REGISTRY[provider_name]
    instance = provider_cls(
        api_key=api_key,
        api_base=api_base,
        model=model,
    )
    _provider_cache[cache_key] = instance
    return instance
```

## 3. Agent 工作流系统

### 3.1 设计思路

**借鉴 RAGFlow 的 `Canvas` + `ComponentBase` 设计模式**，将题库项目中的 AI 操作（题目优化、知识点匹配、题型识别、解析生成等）抽象为可编排的工作流组件。

### 3.2 核心组件

#### 3.2.1 Canvas DAG 执行引擎 `agent/canvas.py`

```python
"""
Canvas DAG 执行引擎

功能：管理组件之间的数据流和执行顺序，支持 DAG（有向无环图）结构
输入参数：dsl（DSL 配置 JSON）、tenant_id
返回值：Canvas 实例
使用场景：所有 AI 工作流的执行引擎

DSL 结构示例：
{
  "components": {
    "begin": {
      "obj": {"component_name": "Begin", "params": {}},
      "downstream": ["llm_0"],
      "upstream": []
    },
    "llm_0": {
      "obj": {"component_name": "LLM", "params": {"prompt": "..."}},
      "downstream": ["switch_0"],
      "upstream": ["begin"]
    },
    "switch_0": {
      "obj": {"component_name": "Switch", "params": {"conditions": [...]}},
      "downstream": ["tool_0", "message_0"],
      "upstream": ["llm_0"]
    }
  },
  "path": ["begin"],
  "globals": {}
}
"""
```

#### 3.2.2 组件基类 `agent/component/base.py`

核心设计（直接借鉴 RAGFlow）：

- `ComponentParamBase` — 参数基类，支持参数校验、更新、序列化
- `ComponentBase` — 组件基类，定义组件生命周期：
  - `invoke(**kwargs)` → 执行组件逻辑
  - `output(key)` / `set_output(key, value)` → 输入输出管理
  - `get_downstream()` / `get_upstream()` → 上下游组件管理
  - `error()` / `exception_handler()` → 异常处理

#### 3.2.3 题库专用组件

| 组件名 | 功能 | 使用场景 |
|--------|------|----------|
| `Begin` | 入口组件，接收输入参数 | 工作流起点 |
| `LLM` | 调用 LLM 进行推理 | 所有 AI 推理节点 |
| `Switch` | 条件分支（根据题型/结果分流） | 不同题型不同处理逻辑 |
| `Loop` | 循环处理（批量题目逐题处理） | 批量操作 |
| `Tool` | 调用外部工具（数据库操作、文件操作） | 读/写题目、知识点 |
| `Message` | 输出结果消息 | 工作流终点 |

#### 3.2.4 题库专用工作流示例

**工作流 1：单题智能处理流水线**

```
Begin → LLM(题型识别) → Switch(题型分支)
  ├─ single → LLM(选项标准化) → Tool(保存题目)
  ├─ fill   → LLM(填空标准化) → Tool(保存题目)
  └─ general → LLM(解答题标准化) → Tool(保存题目)
```

**工作流 2：批量试卷解析流水线**

```
Begin → Loop(逐页处理)
  └─ Tool(OCR页面) → LLM(VLM语义理解) → LLM(题干优化)
      → LLM(知识点匹配) → Tool(保存题目)
```

**工作流 3：智能出卷流水线**

```
Begin → LLM(根据知识点和难度筛选) → Tool(查询题库)
  → LLM(组卷) → Message(输出试卷)
```

## 4. 文档解析管道

### 4.1 设计思路

**借鉴 RAGFlow 的 `Pipeline` 模式**（`rag/flow/pipeline.py`），将当前项目中分散的 PDF 解析、OCR、VLM、LLM 优化等步骤统一为可编排的管道阶段。

### 4.2 当前解析流程（分散式）

```
上传 PDF
  → main.py 接收文件
  → parse_tasks.py Celery 异步任务
  → parse_service.py 编排
      → pdf_service.py 解析（PyMuPDF 提取文本）
      → ocr_service.py OCR 版面分析
      → vlm_service.py VLM 语义理解
      → pdf_service.py LLM 优化题干
      → pdf_service.py 知识点匹配
```

### 4.3 目标解析流程（管道式）

```
上传 PDF
  → Pipeline 入口
  → Stage 1: 文档解析 (PDFParser / MinerUParser)
      → 将 PDF 转为页面图片 + 原始文本
  → Stage 2: OCR 版面分析 (OcrStage)
      → 检测文本块、图形区域坐标
  → Stage 3: VLM 语义理解 (VlmStage)
      → 识别题目内容、题型、图形绑定
  → Stage 4: 题目拆分 (ChunkStage)
      → 按题号切割文本为独立题目
  → Stage 5: LLM 题干优化 (RefineStage)
      → 修正错别字、LaTeX 转换、题型判断
  → Stage 6: 知识点匹配 (KnowledgeStage)
      → 自动匹配/创建知识点
  → Pipeline 输出
```

### 4.4 管道基类设计 `pipeline/base.py`

```python
"""
管道阶段基类

功能：定义管道阶段的统一接口
输入参数：context（管道上下文字典）
返回值：context（更新后的管道上下文）
使用场景：所有管道阶段继承此基类
"""
from abc import ABC, abstractmethod


class StageBase(ABC):
    """管道阶段基类"""

    # 阶段名称
    stage_name: str = ""

    def __init__(self, config: dict = None):
        self.config = config or {}  # 阶段配置

    @abstractmethod
    async def process(self, context: dict) -> dict:
        """处理阶段逻辑

        Args:
            context: 管道上下文，包含上游阶段的所有数据

        Returns:
            更新后的上下文
        """
        ...
```

### 4.5 管道主逻辑 `pipeline/pipeline.py`

```python
"""
文档解析管道

功能：编排多个处理阶段，按顺序执行文档解析流程
输入参数：paper_id / db_session / config
返回值：解析结果
使用场景：试卷上传后的完整解析流程
"""
import logging
from pipeline.base import StageBase


class ParsePipeline:
    """文档解析管道 — 可编排的阶段式处理"""

    def __init__(self, stages: list[StageBase] = None):
        self.stages = stages or []  # 处理阶段列表
        self.context = {}  # 管道上下文
        self.error = ""  # 错误信息

    def add_stage(self, stage: StageBase):
        """添加处理阶段"""
        self.stages.append(stage)

    async def run(self, paper_id: str, db_session, config: dict = None) -> dict:
        """执行管道

        Args:
            paper_id: 试卷ID
            db_session: 数据库会话
            config: 管道配置（可覆盖各阶段默认配置）

        Returns:
            管道执行结果
        """
        # 初始化上下文
        self.context = {
            "paper_id": paper_id,
            "db_session": db_session,
            "config": config or {},
            "stages_completed": [],
        }

        for stage in self.stages:
            logging.info(f"[Pipeline] 开始阶段: {stage.stage_name}")
            try:
                self.context = await stage.process(self.context)
                self.context["stages_completed"].append(stage.stage_name)
                logging.info(f"[Pipeline] 阶段完成: {stage.stage_name}")
            except Exception as e:
                self.error = f"[{stage.stage_name}] {str(e)}"
                logging.error(f"[Pipeline] 阶段失败: {self.error}")
                # 根据配置决定是否继续
                if self.context.get("config", {}).get("stop_on_error", True):
                    break

        return {
            "success": not bool(self.error),
            "error": self.error,
            "stages_completed": self.context["stages_completed"],
            "result": self.context.get("result", {}),
        }
```

### 4.6 阶段实现示例 `pipeline/stages/refine.py`

```python
"""
LLM 题干优化阶段

功能：调用 LLM 修正错别字、转换 LaTeX 公式、判断题型
输入参数：context（包含 questions 列表）
返回值：context（更新后的 questions）
使用场景：管道中的题干优化阶段
"""
from pipeline.base import StageBase
from llm.factory import get_provider


class RefineStage(StageBase):
    """题干优化阶段"""

    stage_name = "refine"

    async def process(self, context: dict) -> dict:
        questions = context.get("questions", [])
        if not questions:
            return context

        # 获取 LLM 适配器
        provider = get_provider(
            provider_name=context.get("config", {}).get("provider", "DeepSeek"),
            model=context.get("config", {}).get("model", "deepseek-chat"),
        )
        if not provider:
            context["_error"] = "无可用 AI 服务商"
            return context

        # 逐题优化
        refined_questions = []
        for q in questions:
            try:
                refined = await provider.refine_questions([q])
                if refined:
                    refined_questions.extend(refined)
                else:
                    refined_questions.append(q)
            except Exception as e:
                refined_questions.append(q)  # 失败则保留原题

        context["questions"] = refined_questions
        return context
```
